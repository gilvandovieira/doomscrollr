# Request: a JSR-native `drizzle-orm` for Deno — with measured memory data

**For the existing Deno-support threads ([#252], [#2440]).** This is a feature request, not a bug —
Drizzle works in Deno through npm/Node-compat. It's about the *cost* of that path and the concrete win
of a JSR-native publish. We bring data: across our dependency set, Drizzle is the single largest
npm-compat memory cost on Deno, by a wide margin.

## Context

Deno's `deno run --watch` retains `npm:` / Node-compat modules across reloads (see Deno [#28107]) — the
old module graph isn't released on restart, so memory ramps every save. That is primarily a Deno issue,
but it makes the *size* of each npm dependency's node-compat footprint visible and painful. JSR-native
modules don't go through that path and don't pay the cost.

## What we measured (Deno 2.8.3)

Per-dependency decomposition — each npm dep **alone** on a `jsr:@hono/hono` base, **no load**, MB of RSS
retained **per `--watch` reload** (restarts verified, ×repeated, stable):

| npm dependency | retained per reload |
|---|---:|
| **`drizzle-orm`** | **+462 MB** |
| `@clerk/backend` | +62 MB |
| `hono` | +19 MB |
| `pino` | +4 MB |
| `postgres` (postgres.js) | +2 MB |

**Drizzle is ~84% of the total.** Note `postgres.js` itself is essentially free — it's not the DB
driver, it's the ORM's module graph through node-compat.

On a **real app with a real Postgres** (matched feed servers, identical query), replacing Drizzle with a
raw JSR Postgres driver (`jsr:@db/postgres`) and moving the rest of the stack to JSR:

| metric (real DB) | npm (Drizzle) | JSR (raw SQL) | Δ |
|---|---:|---:|---|
| warm RSS | 496 MB | 152 MB | **3.3× less** |
| startup peak | 499 MB | 157 MB | 3.2× less |
| RSS under load | 799 MB | 220 MB | 3.6× less |
| `--watch` reload retention | +552 MB/reload | +64 MB/reload | **8.6× less** |
| `/feed` throughput | 2,745 rps | 2,531 rps | ~equal |

It's the loader, not the bytes: the *same library* (Hono) retains **+19 MB/reload via `npm:hono` but
+2 via `jsr:@hono/hono`** — identical code, 10× less, because the JSR path avoids node-compat.

## The ask

A **JSR-native publish of the `drizzle-orm` runtime/ORM core** would let Deno users import it without the
node-compat path, eliminating this retention and startup cost. `drizzle-kit` (the migrator) relies on
Node APIs and is a separate concern — the runtime ORM is what serves requests and what we'd import.

## Feasibility — we read the repo (`drizzle-team/drizzle-orm`, the `drizzle-orm` package)

The runtime ORM reads as close to JSR-ready, especially for Postgres:

- **Pure ESM TypeScript source, zero runtime dependencies.** `"type": "module"`; no `dependencies`
  (every driver — `postgres`, `@neondatabase/serverless`, … — is a *peer* dependency the user supplies).
  No bare `require()`, no `module.exports`, no `__dirname` in `src/`.
- **The PostgreSQL dialect has zero `node:` imports.** `src/pg-core/**` = 0 `node:` imports; the
  `postgres-js` dialect imports only internal modules + the `postgres` peer dep.
- **The little `node:` coupling that exists is isolated and Deno-compatible:** `node:fs`/`node:crypto`
  appear only in `src/migrator.ts` (file-based migrations — separable, and `node:` works on Deno anyway);
  `node:events` only in the MySQL2 / SingleStore sessions. None are on the Postgres serving path.
  `Buffer` (binary columns) works on Deno via `node:buffer` / the global.
- The npm package's dual CJS/ESM output is generated by your build (`scripts/build.ts` + tsup); JSR would
  publish the **TS source directly** — no build step.

So a JSR publish of the runtime ORM looks like a **packaging change, not a runtime rewrite**: resolve the
`~/` path aliases to relative paths, and publish with `--allow-slow-types` (Drizzle's generics will trip
JSR's fast-types check — a documented, supported escape hatch, not a blocker). `drizzle-kit` (the
Node-bound migrator) can stay on npm. For reference, Hono already dual-publishes (`npm:hono` +
`jsr:@hono/hono`) and the JSR path is ~10× lighter on Deno.

Happy to share the full benchmark suite, the per-dependency decomposition scripts, and the matched
real-DB npm-vs-JSR servers so you can reproduce the numbers.
