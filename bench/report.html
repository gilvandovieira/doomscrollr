Memory investigation · Deno 2.8.3

The leak that was three different things.

An API that “locked up the laptop at 3 GB” turned out to be three separate problems. Two were operational. The real runtime bug: `deno run --watch` retains far more memory for `npm:` modules than for JSR across reloads — proven by the same library through two loaders.

2026-06-24i7-12700H · 15 GiB · Linux 7.0.12 Deno 2.8.3 · Node 26.1 · Bun 1.3.14Hono + Drizzle + postgres.js app

00The short version

The “3 GB leak”

Not a code leak — concurrent dev servers stacking (agents + an orphaned test server), each 0.6–2 GB. Fixed a SIGTERM handler that wedged on shutdown.

Production memory

deno compile takes the API from 641 → 226 MB, zero code change — ≈ Node, < ¼ of deno run.

The real runtime bug

--watch retains far more for npm: than JSR — same Hono library: +19 vs +2 MB/reload through the two loaders.

The fix

Dev: external watcher (kill + respawn) or compile. Prod: ship the compiled binary.

02The --watch reload retention

A single deno run --watch dev server grows memory on every save — linearly. On a roomy host it does not plateau: a dead-straight ramp from 143 MB to 2.7 GB over 41 saves, slope unchanged end to end (+65 → +63 MB/save).

Dev loopPer saveAcross a session
deno run --watch, Drizzle (heavy npm)+530 MB→ OOM 226→3,411 MB in 6 saves
deno run --watch, Clerk-floor (minimal npm)+64 MB→ OOM 143→2,740 MB over 41 saves, linear (no plateau)
watchexec -r → fresh deno run (minimal npm)~0flat ~147 MB indefinitely (18 saves, spread 12)
watchexec -r → fresh deno run (heavy npm)~0flat ~340 MB indefinitely
watchexec -r → deno compile binary~0flat ~221 MB

A fresh process per reload is flat — proven with real watchexec -r (§06; respawn ~200 ms). Reclamation is deferred, not absent: a point-in-time gc() of the live isolate doesn’t drop it (it can’t touch a prior run’s retained isolate), and on a host with free RAM nothing forces the runtime to reclaim — so it climbs until OOM. The ~600 MB “plateau” we saw earlier only appears under memory pressure — inside a capped container, where cgroup pressure triggers reclamation. Under concurrent load the loop is also CPU-bound (~6 of 20 cores).

03Root cause — the npm: loader, not the watcher

The decisive control: the same library through two loaders. Hono from npm:hono retains ~+19 MB/reload; from jsr:@hono/hono — identical code — ~+2 MB. Only the npm:/Node-compat resolution path differs.

→ Conclusion

The retained RSS appears tied to the npm/Node-compat module path not being released on --watch restart — native, not V8 old-space (a 256 MB heap cap doesn’t bound it → code-space is the suspect), not glibc-arena. It reproduces in the official denoland/deno:2.8.3 image (stock glibc), so it isn’t our distro. (Mechanism inferred; the npm: ≫ jsr: difference is directly measured.)

This matches Deno #28107 (closed): its repro loads @huggingface/transformers — an npm package — and reaches tens of GB. The ceiling we saw varied by environment (a container reclaimed to ~600 MB; the host kept climbing to ~3.9 GB), not by footprint — so “bigger npm footprint → higher ceiling” is only a qualitative hypothesis for why an ML run reaches that scale.

→ The decomposition that explains it

Each npm dep alone on a JSR base: Drizzle +462, Clerk +62, Hono +19, Pino +4, postgres.js +2 MB/reload. Drizzle is ~84% of the cost — and it's npm-only, so replacing it with raw SQL is the one move that matters. Clerk is the floor (+62, no JSR): the JSR app still climbs ~64 MB/reload, just ~9× slower (3 GB in ~45 reloads vs ~6). postgres.js itself is free — the DB driver isn't the problem; Drizzle's ORM graph through node-compat is.

Do the safe JSR swaps alone help? No — Drizzle dominates

Swapping our app's JSR-available deps (hono, zod, logging) to JSR while keeping Drizzle barely moves the dev-loop ramp — Drizzle's +462/reload is the whole problem:

Deno stackwarm RSS--watch retention
all-npm (Drizzle + npm hono/zod/pino)574 MB+528/save OOM @ 6
safe-JSR only (hono/zod/log → JSR, Drizzle kept)496 MB+510/save OOM @ 6
Kysely + full safe-JSR (+ @db/postgres)158 MB+64/save

The safe swaps alone shave ~14% warm RSS (free) but only ~3% off the reload ramp — it still OOMs in ~6 saves. Drizzle → Kysely is the move that matters; the hono/zod/log → JSR swaps are cheap free-riders you do alongside it, not instead of it. Together they hit the floor: +64/save, 158 MB. Repro: bench/jsr-bench/jsr-safe-run.sh.

Throughput is a wash (the DB query dominates), so the swap is pure upside on memory and startup — at the cost of Drizzle's query builder and an unavoidable Clerk floor.

→ You don’t have to give up the ORM feel — Kysely lands on the floor too

The realistic replacement for Drizzle isn’t raw SQL, it’s Kysely — a pure-ESM, zero-runtime-dep typed query builder, used as a compiler over the same @db/postgres. It keeps compile-time type-safety and still sits on the JSR floor: 152 MB warm, +64 MB/save — identical to raw SQL (+63), ~8× under Drizzle. It adds only ~+1 MB/save over the Clerk floor because almost nothing goes through node-compat (zero runtime deps is why), and it doesn’t even need a JSR publish — npm:kysely is already this clean through the compat loader.

→ The two low-memory paths are alternatives, not additive

Compile only strips deno run's npm-compat overhead — a 2.7× win for npm (639→235) but a 1.5× loss for JSR (153→234), whose run already sits under the binary floor (the two binaries agree at ~234, cross-validating). The lightest config of everything measured is JSR + plain deno run, 153 MB — below both binaries. So: stay on npm/Drizzle → deno compile; go maximalist JSR → just deno run, don't compile. For the JSR stack, compile is a packaging tool (self-contained binary), not a memory tool.

npm deno run warm is core-pinning/warm-up sensitive (639 here vs 496 in §04's matched run); JSR 153 and the ~234 compiled floor are stable. Repro: bench/jsr-bench/compile-compare.sh.

06Why it explodes with coding agents

This is the scenario that started the investigation. The reload retention compounds with process multiplication: each --watch server self-inflates on every save, and multiple agents each run their own server. N agents editing = N processes each ramping linearly.

One agent hammering saves drives a single server up the +64-to-530 MB/save ramp. Three agents editing three servers → three processes each climbing toward 2–3 GB → 6–9 GB and a locked-up laptop, fast. An idle watch server is harmless — no saves, no ramp; agents are the opposite of idle. And it takes only one npm: dependency: Drizzle alone explodes in ~6 saves; even the lone Clerk floor climbs linearly to OOM. It’s the node-compat loader path, not the library — the same Hono retains +19 via npm: but +2 via jsr:.

The fix is a 2×2 — the external watcher kills the ramp, minimal npm lowers the baseline

RSS per save →deno run --watchwatchexec -r (fresh process)
heavy npm — Drizzle+530 → OOM in ~6flat ~340 MB
minimal npm — raw SQL+64 → OOM in ~41flat ~147 MB
minimal npm — Kysely (ORM feel)+64 → OOM in ~41flat ~148 MB

Verified with real watchexec -r over 18 saves: a fresh deno run per change (~200 ms cold start) never stacks — the previous run’s npm-compat layer dies with the process. The two axes are independent and additive: the external watcher flattens the ramp (every row goes flat), minimal npm lowers the baseline (147 vs 340 MB), and Kysely keeps the ORM ergonomics for free (148 MB, identical to raw SQL). The only configs that are actually safe are a fresh process per reload, zero npm deps (pure JSR), or memory pressure that forces reclamation (a capped container). Get Clerk onto JSR and even --watch becomes safe.

A full JSR rewrite would sidestep the watch bug, but Clerk/Drizzle have no JSR equivalents — so for this app the external-watcher dev loop is the practical fix. And if you did go JSR you'd run it, not compile it (§05): JSR + deno run is ~153 MB, below the compiled binary. The runtime decision is otherwise driven by deploy target, not the benchmark.

Methodology & honest caveats

Memory = VmRSS / peak VmHWM from /proc/<pid>/status; server and load generator pinned to separate cores; every run verified a live DB before measuring; Node/Bun ran the unmodified app via a Deno global shim. The --watch finding sharpened through the day — unbounded → graph-size → npm-specific, and the apparent “plateau” proved container-only (linear on a roomy host to ≥2.7 GB) — because each better-controlled experiment refined it; the conclusion here is the one that survived repetition, the heavy-JSR reversal, the stock-glibc container, and a real watchexec -r control.

Single-machine, indicative numbers; relative comparisons are the signal. No secrets, credentials, hostnames, or paths are included. #28107’s open/closed status should be confirmed on GitHub before citing it as a regression. · No production databases were harmed in this investigation (one local dev container was briefly, accidentally, removed by an over-broad docker rm --filter name=t — postgres contains a t — and restored from its volume).
